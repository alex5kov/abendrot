2. Add Security Constraint

    • Maybe we do not want everybody to access this spy thingy. Let's implement security : only registered Sap Community Network members (well, that's a lot ... ) will be able to access it.

Let's implement standard servlet security as defined in the spec.

Create a new web.xml file in the src/main/webapp/WEB-INF directory. Copy & paste the code from this version from github. 

<login-config>
  <auth-method>FORM</auth-method>
</login-config>

<security-constraint>
  <web-resource-collection>
    <web-resource-name>Protected Area</web-resource-name>
    <url-pattern>/spy.jsp</url-pattern>
  </web-resource-collection>
  <auth-constraint>
    <role-name>Everyone</role-name>
  </auth-constraint>
</security-constraint>

<security-role>
  <description>All SAP NetWeaver Cloud users</description>
  <role-name>Everyone</role-name>
</security-role>

In Eclipse, in the servers view, double click on the Local Test Server for SAP NetWeaver Cloud to open the server configuration page, select the 'Users' tab, then click New User.

You are free to enter whatever content in the following dialog box to create a new user. Close the dialog, and do not forget to save your modifications of the server configuration. (CTRL-S)

Run spy.jsp in the Local Test Server for SAP NetWeaver Cloud. Bing, an authentication form is displayed. Enter the Credentials for the user you've just created, and you are in.

This local user management provides a nice de-coupling of the development cycle from 'true' authentication providers, such as LDAP servers, that maybe more complex to configure, and less performant, or even out of reach in certain circumstances, e.g. working off-line. It is nevertheless possible to plugin locally other authentication providers. I leave that as an exercice for the reader.

Let's see what happens in the cloud. Deploy this application with the modified (new, actually) web.xml. Again, you should be familar with the process now, but I repeat it - for the last time, promised:

  • commit to local git
  • push to remote git
  • mvn clean deploy
  • grab another cup of
  • browse to https://<your app><your user ID>trial.nwtrial.ondemand.com/<your war name>/spy.jsp

Bing, another authentication form is displayed. Enter now your SCN Credentials, and you are in.

Authentication on the cloud involves the Security Assertion Markup Language 2.0 (SAML 2.0) protocol.

Your application subcontracts authentication to the SAP Community Network (cf. the URL in the screen shot above : https://accounts.sap.com/...).

For this, a bi-directionnal trust relationship must exist between your app and SCN: your app trusts that the authentication token provided by SCN truly identifies you, but before, SCN has trusted your app to be entitled to query - and make good usage - of authentication tokens. Fortunately, SCN and NetWeaver Cloud are managed by the same guy (SAP), which helps fostering a strong trust relationship.

For SAP NetWeaver Cloud customers that have their own authentication provider. e.g. corporate LDAP, it is possible to setup the same trust relationship and communication process. It is as well possible for the authentication provider of customer to pass extended attributes to the application, such as group information (e.g. admins, super-users, users, guests), upon which the application can react accordingly.

However, to my knowledge, there is currently no way to subcontract the authentication to 3rd party identity providers, such as Facebook, Twitter, etc. There is little doubt that this feature will be heavily requested.

3. Get Back Tomcat in the Game

    • Tomcat standalone server needs a little uplift to be able to handle authentication. Only for Windows. Sob.

You may have noticed that the previous section did not show how to run your application in Tomcat, now that servlet security is enabled.

Unfortunately, with this instruction in the web.xml ... 

<login-config>
  <auth-method>FORM</auth-method>
</login-config>
... the servlet container expects a HTML form to be implemented somewhere to handle the authentication. SAP NetWeaver Cloud local as well as remote systems provide a default HTML form (we've seen both). However, Tomcat has no default. Trying to run spy.jsp on Tomcat leads to a miserable "No login page was defined for FORM authentication" error message:

At least the error message is clear. We are left with these options:

  1. Implement such a FORM. Not trivial and above all, the application code will diverge between Tomcat and SAP NetWeaver Cloud.
  2. Comment and un-comment the security section in web.xml at each switch from testing with Tomcat to testing with SAP NetWeaver Cloud. I even did that for some time. No comment.
  3. As a variation of 2., implement maven resources filtering or plugin to rewrite the web.xml file at build time based on some profile or parameter. Tecnically feasable, but a pain in the neck.
  4. Stop using Tomcat. But we need Tomcat ! First to ensure that our application is not overly coupled with SAP NetWeaver Cloud, second, because hot reloading of jsp is unvaluable to speed up development, and I did not find how to get hot reload with SAP NetWeaver Cloud local test server. If you know that it is possible, please wave your hand.

Enter WAFFLE, open source Windows & Active Directory Authentication Framework for C# and Java.. Download latest version from here, uzip it it somewhere and copy ... 

jcl-over-slf4j-1.7.2.jar
jna-3.5.0.jar
platform-3.5.0.jar
slf4j-api-1.7.2.jar
waffle-jna.jar
waffle-tomcat7.jar
... from <waffle>/Bin to <apache-tomcat>/lib.

Open context.xml in Package Explorer > Servers > Tomcat v7.0 Server:

Add 

<!-- https://github.com/dblock/waffle/blob/master/Docs/tomcat/TomcatSingleSignOnValve.md -->
<Valve className="waffle.apache.NegotiateAuthenticator" />
<Realm className="waffle.apache.WindowsRealm" />

to install the waffle authentication provider for Windows in Tomcat. Stop and restart tomcat, run spy.jsp, and bingo !

See here for more options to configure Tomcat with waffle.

I am afraid that Linux and Mac developers will have to opt for one of the options 1. to 4. above, option 3. being probably the best. If you have a better solution, please contribute to this tutorial, it's on Github .
